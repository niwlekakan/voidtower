## 4. AI-reachable ingress points (exhaustive, verified in source)

This is the most safety-critical section. Every claim below cites the exact call site, re-verified directly against source for this regeneration. **Both gaps prior maps tracked as open are now CLOSED in `main`**: the scope-bypass gap closed with P0-06 (unchanged this cycle, re-verified below), and P0-04's hardcoded irreversibility denylist merged this cycle (rewritten below).

### The voidwatch choke point (P0-01 → P0-02 → P0-03) — unchanged this cycle

`voidwatch::evaluate(db, actor, action_kind, action, resource) -> Verdict` (`voidwatch/mod.rs:132-159`) is the sole caller of `policy::check` from the three AI/automation-reachable ingress files. Enforced by a source-grep regression test (`voidwatch/mod.rs:533-549`) that fails the build if `api/mcp.rs`, `api/studio.rs`, or `api/integrations.rs` ever call `policy::check(` directly. Verified as true today for all three files; no diff to any of these files since the prior map.

**P0-01** was a no-op-by-default choke point: `Read`-classified actions always `Allow`; `Mutating` actions got exactly the same `policy::check` verdict as before.

**P0-02 ended the no-op.** `policy.rs`'s `check()` now default-**denies** when no `policy_rules` row matches, for `actor_type` in `api_token` / `automation` / `ai` (`user` stays RBAC-governed, default-allow). Escape hatch: `voidwatch_default_allowlist` (§3).

**P0-03 adds a pre-pass in front of both of the above** (`mode_pre_pass`, `voidwatch/mod.rs:165-193`), evaluated first, for *every* actor class equally (ADR-004 constraint 2 — the ladder isn't AI-only):
- Looks up the current `mode::Mode` for the resource's scope (falling back to the `"global"` scope row, §3). **If nothing has ever been configured — the default state of every install today, since no HTTP handler can set a mode yet — the pre-pass is a complete no-op and falls straight through to P0-01/P0-02 behavior.** This is why every ingress point below still behaves identically to the P0-02-era description in practice, despite the new code being live.
- `Observer` → `Deny` unconditionally. `Assisted` → `RequireApproval` unconditionally. `Yolo` → `Allow` unless the action's `risk_class::for_action()` is `Irreversible`, in which case `RequireApproval` — this is the mode-ladder mechanism the P0-04 denylist coordinates with (see below). `Trusted` → `RequireApproval` for `Destructive`/`Irreversible`-risk actions always; for `Mutate`-risk actions, `Allow` only if `(actor_type, action, resource_type)` is on `voidwatch_default_allowlist` — allowlisted actions against a `SNAPSHOT_CAPABLE_RESOURCE_TYPES` resource (`vm`/`container`/`app`) get `AllowRequireSnapshot` instead of a plain `Allow`.
- `Verdict::AllowRequireSnapshot(String)` and `Verdict::RequireApproval(String)` are terminal states neither ingress point below has anywhere to act on — no approvals queue exists — so both call sites treat every non-`Allow` verdict as a block (fail closed), per `integrations.rs:726-735`'s `verdict_block_reason()`.

**Practically, as of this map:** unless an operator has explicitly configured a mode (no UI/API path to do so exists yet), every ingress point below behaves exactly as the P0-02-era map described. The mode ladder is live code, correctly tested, but dormant until wired to an operator-facing control.

### Risk classification (P0-03, `voidwatch/risk_class.rs`) — reclassifications this cycle

Two independent tables (§1) — `for_action()` is the one the mode ladder above actually consults; `ROUTE_RISK_CLASSES` is a separate, exhaustively-tested per-route ledger, now consumed as the classification ledger `denylist.rs` reconciled its 8 items against. **`for_action()`'s fail-safe default is still `Irreversible` for any action name it hasn't explicitly classified** — this remains the only enforcement of irreversibility-style caution reachable through the AI/automation ingress vocabulary, distinct from the route-level denylist below, and only takes effect once a mode is configured (per the dormancy note above).

**Three `ROUTE_RISK_CLASSES` entries were reclassified `Destructive` → `Irreversible` this cycle** (verified in `git diff ae2095c..HEAD -- backend/src/voidwatch/risk_class.rs`): `POST /api/apps/:project_name/delete-volumes`, `POST /api/apps/:project_name/purge` (both ADR-004 item 11, a documented judgment call — no literal `keep_data` flag exists in source), and `DELETE /api/proxmox/:host_id/vms/:vmid/snapshot/:snapname` (ADR-004 item 10, resolved this cycle — see the map preamble).

### The irreversibility denylist (P0-04) — merged this cycle (`backend/src/voidwatch/denylist.rs`)

### The scope-bypass gap — CLOSED in `main` (P0-06, merged `7fa1b77`) — unchanged this cycle

Verified directly against `backend/src/auth/scope_enforce.rs`, `backend/src/api/bearer_auth.rs`, and `backend/src/api/mod.rs`'s layer ordering (§1/§2/§6); no diff to any of these files since the prior map. The original bug, for reference:

1. `bearer_auth::middleware` is mounted globally. A request with `Authorization: Bearer <token>` and no session cookie calls `auth::validate_api_token_any()` — checked only that the token hash exists and wasn't expired, never scopes — and minted a full 1-hour session via `auth::create_temp_session()` for the token's owning user.
2. Every downstream handler's `require_user`/`require_admin`/`require_owner` helper then saw an ordinary session and applied the token owner's actual role, not the token's declared scope.
3. `ALL_SCOPES` (`integrations.rs:29-55`) was consulted in exactly two places in the whole codebase. Every other scope-implying endpoint performed no scope check at all.
4. Net effect: a token minted with only `metrics:read`, if owned by an admin/owner user, could call any admin/owner-gated endpoint.

**The fix (P0-06, ADR-003):** `bearer_auth::middleware` resolves the token's declared scopes and attaches them via a `TokenScopes` extension. `auth::scope_enforce::middleware`, mounted directly after it, is the sole choke point ADR-003 specified. It is a structural no-op for any request without `TokenScopes` (proven by `human_session_cookie_login_is_unaffected_by_scope_changes` in `api/scope_bypass_tests.rs`). For token-originated requests, `ROUTE_SCOPES` (`scope_enforce.rs:55-156`) is deny-by-default: a route with no entry is closed to tokens regardless of the owning user's role. `scope_enforce` is mounted on `main_router` only, not `embed_router` (§6).

**Capability tiers:** `integrations::CAPABILITY_TIERS` (§2) gives `POST /api/integrations/tokens` an optional coarser `tier` shortcut on top of the scope enforcement above; `admin-never` mints an empty scope set, blocked structurally by the same default-deny table.

### Secrets redaction on the AI context path (P0.5, `api/redact.rs`, declared as `api::mcp::redact`) — unchanged this cycle

New defense-in-depth layer, independent of and downstream from the voidwatch gating above — it runs on the *response*, not the request, and applies regardless of verdict. Two layers (module doc comment, `redact.rs:6-16`): (1) `redact_known_values` — exact-match redaction of every currently-stored `secrets` table value; (2) `redact_patterns` — heuristic redaction of secret-*shaped* substrings not registered as VoidTower secrets at all. `redact_for_ai(state, text)` is the convenience entry point. **Wired into exactly three places**:
- `api/mcp.rs`'s `invoke_tool()` (`mcp.rs:573`) — every MCP tool-call result, before it's returned.
- `api/studio.rs`'s `mcp_invoke` — inherited automatically by reusing `mcp::invoke_tool` (§2).
- `api/ai_context.rs`'s `get_context` (`ai_context.rs:159`) — the serialized context bundle.

Not wired into `api/ai_ask.rs` (`POST /api/ai/ask`) directly.

## 6. Observed code conventions

- **Auth pattern (repeated per-file, not shared middleware):** nearly every `api/*.rs` file defines its own local `require_user`/`require_admin`/`require_owner` async fn that reads the `vt_session` cookie from `CookieJar`, calls `auth::validate_session`, and maps `None`/error to `AppError::Unauthorized`/`Internal`. Role checks are inlined per-handler rather than expressed as a reusable Axum extractor. `members.rs`/`node_enroll.rs` follow this same copy-per-file pattern.
- **Global middleware layering:** `main_router` has **four** cross-cutting concerns implemented as actual Axum middleware: `security_headers`, `demo_guard::middleware`, `scope_enforce::middleware`, and `bearer_auth::middleware`, layered in that source order (`mod.rs:450-459`) — `.layer()` wraps outward, so the **last**-added layer is outermost and runs first on a request: `bearer_auth` (outermost/first) → `scope_enforce` → `demo_guard` → `security_headers` → `cors` (innermost, closest to the router). `embed_router` only ever gets `cors` + `bearer_auth`. CORS is permissive (`Any`/`Any`/`Any`). Unchanged this cycle.
- **`#[rustfmt::skip]` to protect a hand-aligned table from `rustfmt`'s `mod`-graph cascade:** `router()` in `api/mod.rs` is annotated `#[rustfmt::skip]` (`mod.rs:104`) because `rustfmt`, invoked on this file (or transitively via `main.rs`, the crate root), treats every file reachable through `mod` declarations as one formatting unit and will silently reflow all of them. This only protects `router()`'s own table — it does not prevent an out-of-scope `cargo fmt --all` from reformatting every other file in the crate. `CLAUDE.md` was hardened this cycle (`ed09406`) with an explicit pre-commit self-check step (`git diff --stat main...HEAD`, read the file list, every file must be one you meant to touch) specifically targeting this recurring failure mode — a harness/process change, not a code convention, but directly downstream of this one.
- **Error handling:** single `AppError` enum via `thiserror`, with `#[from]` conversions for `sqlx::Error` and `anyhow::Error`; handlers return `Result<Json<T>>` and propagate with `?` or explicit `.map_err(...)`.
- **DB access:** raw `sqlx::query`/`query_as::<_, T>` with `#[derive(sqlx::FromRow)]` structs defined inline in the handler file (not a separate repository/DAO layer).
- **Audit logging:** mutating human-initiated actions call `audit::log(...)` (`backend/src/audit/mod.rs:20-52`, §1); AI/automation-sourced actions use the sibling `audit::log_sourced(...)` with an extra `source` param (e.g. `"odysseus"`).
- **Actor detection for policy/voidwatch:** the `MaybeTokenActor` Axum extractor (`policy.rs`) reads an `ApiTokenActor` marker inserted by `bearer_auth::middleware`; ingress handlers construct a `voidwatch::Actor{ kind }` explicitly per call site rather than deriving it generically — opt-in/explicit per-handler, not automatic.
- **Naming:** snake_case DB columns/Rust fields, kebab-case-ish REST paths, route path segments named after the resource (`:id`, `:name`, `:project_name`, `:host_id`, `:vmid`, `:user_id`, `:drive_id`, `:app_id` depending on domain).
- **Test conventions:** `#[test]`/`#[tokio::test]` appear in `backend/src/api/tabs.rs`, `backend/src/api/agents.rs`, `backend/src/api/scope_bypass_tests.rs`, `backend/src/voidwatch/mod.rs` (17 tests including the mode-ladder acceptance suite), `backend/src/voidwatch/mode.rs`, `backend/src/voidwatch/risk_class.rs`, `backend/src/voidwatch/allowlist_seed.rs`, `backend/src/voidwatch/denylist.rs` (**new this cycle** — 11 named tests, see §4), `backend/src/policy.rs`, `backend/src/api/integrations.rs`, and `backend/src/api/mcp.rs`/`redact.rs`/`studio.rs` (redaction corpus tests). No established test-file-naming convention beyond `#[cfg(test)] mod tests { ... }` inline at the bottom of the same file as the code under test. A recurring pattern: since `db::run_migrations` doesn't create `policy_rules`/`voidwatch_default_allowlist`/`voidwatch_mode_settings`, test setup functions create all three directly with inline `CREATE TABLE` statements — `voidwatch::tests::create_policy_tables` is `pub(crate)` specifically so sibling test modules (including `denylist.rs`'s, this cycle) can reuse it. Also: `backend/tests/schema_golden.sql` plus `db/mod.rs` tests diffing live schema and proving upgrade-safety.
- **Parser-based exhaustiveness tests:** `risk_class::tests::every_registered_route_has_a_risk_class` parses `api/mod.rs` with a manual balanced-paren scanner to extract every registered `(method, path)` and asserts a bijection against `ROUTE_RISK_CLASSES`; `voidwatch::mod::tests::evaluate_is_the_only_caller_of_policy_check_from_ai_ingress` is the substring-based precedent it extends. Same spirit as `db/mod.rs`'s schema-golden test.
- **Directory-tree-walk invariant tests (new this cycle, `denylist.rs`):** `docker_sock_no_bypass_invariant` (§1/§4) is a new variant of the parser-based-exhaustiveness pattern above — instead of parsing one file, it recursively walks every `.rs` file under `backend/src` via `std::fs::read_dir` and asserts a fixed, sorted list of call sites for a specific symbol (`bollard::Docker::connect_with_unix_defaults`), excluding its own file by name so the test's doc comments/assertion strings naming the symbol don't self-trigger. Worth reusing for any future "assert exactly these files call X" invariant that can't be expressed as a single-file source-grep.
- **Shared-reasoning-helper-to-prevent-drift:** `integrations.rs`'s `verdict_block_reason()` (`integrations.rs:726-735`) exists because two near-identical inline `match` arms at separate call sites drifted apart. Worth applying proactively anywhere near-identical verdict/permission-handling logic is about to be copy-pasted to a second call site. **The P0-04 denylist follows the same instinct at the ledger level**: rather than let `denylist.rs`'s route list and `risk_class.rs`'s classifications drift independently, `assert_item_routes_are_irreversible` ties every denylist item's routes to a matching `ROUTE_RISK_CLASSES` entry as a build-time-enforced invariant, not just a one-time manual cross-check.
- **`test_support.rs` as a per-module shared-fixture file:** `api/mcp.rs` declares `pub(crate) mod test_support;` (backed by `api/test_support.rs`, `#![cfg(test)]`-gated) providing a shared `AppState` builder and session/DB setup helpers.
- **Secrets:** encrypted with AES-256-GCM using a 32-byte key generated/stored at `<config_dir>/secrets.key` (0600 perms); the same decrypt routine is duplicated in `api/proxmox.rs` rather than shared — still a candidate for consolidation but not something to change without a task naming it.
- **CLI subcommands** (`main.rs`) exist in parallel to the HTTP API for `user` and `backup` management, sharing the same `backend::auth`/`backend::backups` functions as the web handlers.
- **Core-logic-vs-HTTP-handler split:** `wireguard.rs`'s `add_peer`/`delete_peer` handlers were split into thin HTTP wrappers around `pub(crate) add_peer_core`/`delete_peer_core` so `node_enroll.rs` could reuse the exact provisioning logic without duplicating it or going through an HTTP round-trip.
- **Frontend:** one shared typed API client per client surface (`frontend/src/api/client.ts`, `mobile/src/api/client.ts` as an explicit port of it) namespaced by backend domain, `credentials: 'include'` (web) on every request; `frontend/`'s client additionally branches on `isTauri()`. Two parallel UI shells exist within `frontend/` (`frontend/src/pages/*` classic pages and `frontend/src/aios/panels/*`), plus two independent client trees (`frontend/src-tauri/`, `mobile/`) that are thin/native wrappers rather than forks of the React app.
