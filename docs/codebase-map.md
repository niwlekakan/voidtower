# VoidTower Codebase Map

Regenerated by a read-only survey of this repo at the current `main` HEAD, cross-checked line-by-line against source, superseding the map committed at `7b2e568`. This map is descriptive, not authoritative — verify behavior in source before relying on it for policy/auth/schema work. All paths are relative to the repo root unless stated otherwise.

Since the prior map: P0-01 landed the `voidwatch` policy choke point (§4), and three features shipped — tiered accounts + mandatory MFA (`guest`/`demo`/`member` roles), WireGuard fleet node enrollment, and a self-hosting hub for the new `member` role — plus a Tauri desktop wrapper and an Expo mobile app alongside `frontend/`. Corrections to the prior map's own gaps are called out inline where found (it had omitted `backend/src/audit/mod.rs` from §1 entirely).

| `backend/src/networking/mod.rs` | Stub only — comment describing planned interface/route/firewall introspection. **Actual network + firewall logic lives in `api/network.rs` and `api/firewall.rs`.** |
| `backend/src/oidc.rs` | OIDC/Authentik SSO: loads per-instance `OidcSettings` from DB, PKCE flow via the `openidconnect` crate, role mapping from claims. Unchanged since the prior map; still a forbidden-zone file. |
| `backend/src/policy.rs` | The AI/automation policy engine: `PolicyRule` struct, `check()` function (default-allow when no rule matches), `ApiTokenActor`/`MaybeTokenActor` extractor. **Unchanged since the prior map** (verified: no diff against `7b2e568`) — forbidden zone, and P0-01 deliberately did not touch it; `voidwatch.rs` (new, see below) wraps it instead. |
| `backend/src/security/mod.rs` | Stub only — comment describing planned security-scan checks. **Actual session-revocation logic lives in `api/security.rs`.** |
| `backend/src/services/mod.rs` | systemd wrapper: `ServiceInfo`, `ServiceAction` enum (Start/Stop/Restart/Enable/Disable), `is_systemd_available()`, shells out to `systemctl`. |
| `backend/src/storage/mod.rs` | Block device / mount / RAID / SMART introspection structs (`BlockDevice`, `MountInfo`), used by `api/storage.rs`. |
| `backend/src/terminal/mod.rs` | PTY handling via `portable-pty` for the in-browser terminal (`ClientMessage`/`ServerMessage` WS protocol), used by `api/terminal.rs`. |
| `backend/src/vms/mod.rs` | Stub-ish: only `is_libvirt_available()` implemented; comment describes planned `list_vms`/`vm_action`. Local KVM + Proxmox VM logic actually lives in `api/vms.rs` and `api/proxmox.rs`. |
| `backend/src/voidwatch.rs` | **New (P0-01).** The single choke point AI/automation-reachable mutating actions must pass through before reaching `policy::check` (EDD §3.2, gap-analysis P0.1). `evaluate(db, actor: Actor, action_kind: ActionKind, action, resource: Resource) -> Verdict` — `Read` actions always `Allow` without consulting `policy_rules`; `Mutating` actions call `policy::check` exactly as before. Per its own doc comment and a compile-time-ish regression test (`voidwatch.rs:265-279`, greps the three ingress files' source for a literal `policy::check(` call), this is explicitly a **no-op-by-default choke point** (ADR-001): identical verdicts to pre-P0-01 behavior, its only effect is structural — `api/mcp.rs` and `api/studio.rs`'s `mcp_invoke` now consult `policy_rules` at all, where before they never did. Default-deny (P0-02) and the risk-class/mode-ladder table (P0-03) are explicitly out of scope and not yet implemented (both are in `.devteam/failed/`, see §4). |

**Convention note:** several top-level modules (`alerts`, `automation`, `cluster`, `networking`, `security`, and partially `vms`) are comment-only placeholders from an earlier phased plan; the real, shipped logic for those feature areas lives entirely in the corresponding `backend/src/api/*.rs` file. `backend/src/audit/mod.rs` is the one exception that looks like it could be a stub-vs-handler split but isn't — it's genuinely load-bearing shared code. New agents should not assume a top-level module's existence implies (or disproves) working logic — verify against the `api/` file and the module body itself.

| `nodes` (`db/mod.rs:~187`) | **New.** Enrolled fleet devices (phones/tablets/pis): `display_name`, `device_type`, `owner_user_id`, `wg_peer_id`/`wg_public_key` (its WireGuard identity), `token_hash` (heartbeat-only bearer token, **not** an `api_tokens` row), `last_seen`/`last_telemetry`, `agent_capable`, `approved`. |

## 4. AI-reachable ingress points (exhaustive, verified in source)

This is the most safety-critical section. Every claim below cites the exact call site. **This section changed substantially since the prior map** — P0-01 (`voidwatch.rs`, §1) landed and closed the single largest gap the prior map documented; a second, still-open gap (bearer-token scope enforcement) has an accepted ADR but no implementation yet.

### The voidwatch choke point (P0-01) — what it actually changed

`voidwatch::evaluate(db, actor, action_kind, action, resource) -> Verdict` (`voidwatch.rs:82-105`) is now the sole caller of `policy::check` from the three AI/automation-reachable ingress files. This is enforced by a source-grep regression test (`voidwatch.rs:265-279`) that fails the build if `api/mcp.rs`, `api/studio.rs`, or `api/integrations.rs` ever call `policy::check(` directly again. Verified as true today for all three files.

Per its own doc comment and ADR-001's sequencing clarification, **this is a no-op-by-default choke point**: `Read`-classified actions always `Allow` without consulting `policy_rules` at all; `Mutating` actions get exactly the same `policy::check` verdict as before P0-01 (still default-allow when no rule matches — that flip is P0-02, not yet implemented, see below). The behavioral change is purely structural: two ingress points that previously called `policy::check` **zero times** now call it every time a mutating action is attempted.

### Ingress points and their current gating status

**`backend/src/api/mcp.rs` — MCP server (`GET /api/mcp`, `POST /api/mcp/message`)**
- Auth: `check_mcp_auth()` requires `odysseus.mcp_enabled` setting = `"true"` and a valid Bearer token via `auth::validate_api_token_any` (unscoped — any valid token from any user works).
- `tools/call` dispatch now routes through `invoke_tool()` (`mcp.rs:441-...`), which classifies each tool via `tool_action_kind()`/`READ_ONLY_TOOLS` (`mcp.rs:426-441`: `list_nodes`, `get_node_metrics`, `list_containers`, `list_services`, `list_alerts`, `get_container_logs`, `list_routes`, `read_file`, `search_code`, `get_template` — still the only 10 tools registered, still all read-only) and calls `voidwatch::evaluate(db, Actor{ApiToken}, action_kind, name, Resource{"mcp_tool", name})` before dispatch. Any tool not in `READ_ONLY_TOOLS` is treated as `Mutating` by default — i.e. a future write tool added here without also being added to that list is gated by default (deny-capable), not silently ungated as before. This is the exact regression the code comment at `mcp.rs:428-432` calls out.
- `tool_read_file`/`tool_search_code` (delegating to `ai_context.rs`) still do path-traversal sanitization but otherwise expose the entire repo tree read-only to any valid token holder — unchanged, and still `Read`-classified (correctly, since it's read-only).

**`backend/src/api/studio.rs` `mcp_invoke` (`POST /api/studio/mcp/invoke`)**
- Auth: `require_user()` (session cookie only) — the in-UI "Studio" panel's tool-test button.
- Now calls `super::mcp::invoke_tool(&state, Actor{ kind: ActorKind::User }, &req.name, req.arguments)` (`studio.rs:563`) — the same gated entry point `mcp.rs` uses, reusing the identical tool set and now the identical voidwatch check. Per `voidwatch.rs:19-22`, `ActorKind::User` intentionally matches only `"*"`-scoped policy rules, preserving the existing convention that session-authenticated humans are governed by RBAC (`auth::User::role`), not `policy_rules`, elsewhere in the codebase.

**`backend/src/api/integrations.rs` `manifest` (`GET /api/integrations/odysseus/manifest`)**
- Unchanged: no auth check at all, purely a JSON tool-surface description consumed by Odysseus, does not itself execute anything. The real ingress points it documents are analyzed below and under "the scope-bypass gap."

**`backend/src/api/integrations.rs` `webhook` (`POST /api/integrations/webhooks`)** — gated, and now more completely than before:
- Auth: constant-time comparison against `odysseus.webhook_secret`, plus `odysseus.enabled`/`emergency_disabled` checks.
- Structured resource actions (`container.restart/start/stop`, `service.restart/start/stop`) call `voidwatch::evaluate` (`integrations.rs:859-875`, actor kind `Automation`) — same as before, just re-pointed from `policy::check` to `voidwatch::evaluate`.
- **The `automation_id`-triggered path is now also gated** (`integrations.rs:677-748`, factored into `run_automation_job()`): it now calls `voidwatch::evaluate(db, Actor{Automation}, Mutating, "automation.run", Resource{"automation_job", automation_id})` **before** running `job.command` via `bash -c`, and returns `AppError::PolicyDenied` (logging a `"blocked"` audit entry) on a `Deny` verdict. **This closes the prior map's largest documented finding** — previously this path called `policy::check` zero times and any user with the shared webhook secret could execute any pre-registered automation job's arbitrary shell command with zero policy involvement. It is still true that any authenticated non-viewer UI user can create arbitrary-command automation jobs via `POST /api/automation` (`api/automation.rs`) — that entry point is unaffected and out of scope for voidwatch (it's a human, session-authenticated action, not AI/automation-originated).

**`backend/src/api/ai_ask.rs` (`POST /api/ai/ask`)** — unchanged: session-cookie-only, human-initiated chat assistant; documents an external MCP tool server (`odysseus-mcp-servers/voidtower_server.py`) whose write tools call the same underlying REST endpoints, so the real ingress control point for those is the REST handler's own auth (see the scope-bypass gap below), not this file.

**`backend/src/api/webhooks.rs` (`/api/webhooks*`)** — unchanged: outbound notification-webhook system, admin-gated, not an AI ingress point; do not confuse with `integrations.rs`'s inbound `webhook` handler.

**`backend/src/api/ai.rs`, `ai_context.rs`, `ai_providers.rs`, `agents.rs`** — unchanged from the prior map's analysis: admin/session-gated configuration or read-only surfaces, not newly-relevant to voidwatch.

### The scope-bypass gap (still open — accepted ADR, not yet implemented)

The prior map's second finding — upstream of `policy.rs`/`voidwatch.rs` entirely — is **unchanged in source** as of this map (verified: no diff to `bearer_auth.rs` or the relevant `auth/mod.rs` sections since `7b2e568`, and `backend/src/auth/scope_enforce.rs` named in ADR-003's grant does not exist in the tree):

1. `bearer_auth::middleware` (`bearer_auth.rs:10-27`) is mounted globally. A request with `Authorization: Bearer <token>` and no session cookie calls `auth::validate_api_token_any()` — checks only that the token hash exists and isn't expired, **never scopes** — and mints a full 1-hour session via `auth::create_temp_session()` for the token's owning user.
2. Every downstream handler's `require_user`/`require_admin`/`require_owner` helper then sees an ordinary session and applies the token owner's actual role, not the token's declared scope.
3. `ALL_SCOPES` (`integrations.rs:29-55`) is consulted in exactly two places in the whole codebase (`events.rs`, `integrations.rs`'s own `event_stream`). Every other scope-implying endpoint performs no scope check at all.
4. Net effect is unchanged from the prior map: a token minted with only `metrics:read`, if owned by an admin/owner user, can call any admin/owner-gated endpoint.

**What's new:** `docs/adr/ADR-003-*.md` is **Accepted** (signed 2026-07-09), authorizing a fix (a new mandatory scope-enforcement middleware plus a route→required-scope table, granting `backend/src/auth/mod.rs`, `backend/src/api/bearer_auth.rs`, a new `backend/src/auth/scope_enforce.rs`, and `backend/src/api/integrations.rs`). The corresponding task, `.devteam/failed/P0-06-scope-token-and-bearer-auth-bypass.md`, is in `.devteam/failed/` (not merged) — **the ADR being Accepted is authorization to implement, not evidence the fix landed; it has not landed.** A task consuming this map for P0.6 work should verify `.devteam/failed/`, `.devteam/blocked/`, and `.devteam/queue/` current contents directly rather than trusting this snapshot, since queue state changes independently of source.

Also newly relevant to this gap: `node_enroll.rs`'s heartbeat token (§2/§3) was deliberately designed **not** to be an `api_tokens`-table token specifically because of this bypass — see the doc comment at `node_enroll.rs:64-68` — so the fleet-enrollment feature is not itself a new instance of the gap, it was built defensively around the known-open one.

## 6. Observed code conventions

- **Auth pattern (repeated per-file, not shared middleware):** nearly every `api/*.rs` file defines its own local `require_user`/`require_admin`/`require_owner` async fn that reads the `vt_session` cookie from `CookieJar`, calls `auth::validate_session`, and maps `None`/error to `AppError::Unauthorized`/`Internal`. Role checks are inlined per-handler rather than expressed as a reusable Axum extractor. **The two new handler files (`members.rs`, `node_enroll.rs`) follow this exact same copy-per-file pattern** rather than introducing a shared helper — consistent with the existing convention, not a new one.
- **Global middleware layering:** now **three** cross-cutting concerns are implemented as actual Axum middleware (was two): `security_headers`, `demo_guard::middleware` (new), and `bearer_auth::middleware`, layered in that source order (`mod.rs:430-432`) — `.layer()` wraps outward, so the **last**-added layer is outermost and runs first on a request: `bearer_auth` (outermost/first) → `demo_guard` → `security_headers` → `cors` (innermost, closest to the router). This ordering is load-bearing: a bearer-token request has its temp session cookie minted by `bearer_auth` before `demo_guard` ever inspects the request for a `role = "demo"` session. CORS is permissive (`Any`/`Any`/`Any`).
- **Error handling:** single `AppError` enum via `thiserror`, with `#[from]` conversions for `sqlx::Error` and `anyhow::Error`; handlers return `Result<Json<T>>` and propagate with `?` or explicit `.map_err(...)`.
- **DB access:** raw `sqlx::query`/`query_as::<_, T>` with `#[derive(sqlx::FromRow)]` structs defined inline in the handler file (not a separate repository/DAO layer) — the new `members.rs`/`node_enroll.rs` continue this (e.g. `NodeRow` in `node_enroll.rs:74-85`, `DriveSummary` in `members.rs:71-79`).
- **Audit logging:** mutating human-initiated actions call `audit::log(...)` (`backend/src/audit/mod.rs:20-52`, §1); AI/automation-sourced actions use the sibling `audit::log_sourced(...)` with an extra `source` param (e.g. `"odysseus"`). Every new `members.rs`/`node_enroll.rs` mutating handler follows this convention (e.g. `members.access.grant`, `nodes.enroll`, `nodes.pairing_code.create`).
- **Actor detection for policy/voidwatch:** the `MaybeTokenActor` Axum extractor (`policy.rs`) reads an `ApiTokenActor` marker inserted by `bearer_auth::middleware`; ingress handlers construct a `voidwatch::Actor{ kind }` explicitly per call site (`ActorKind::ApiToken` in `mcp.rs`'s bearer-token dispatch, `ActorKind::User` in `studio.rs`'s session-authenticated path, `ActorKind::Automation` in `integrations.rs`'s webhook paths) rather than deriving it generically — this is opt-in/explicit per-handler, not automatic, same spirit as the pre-existing `if is_token { policy::check(...) }` pattern in `containers.rs`/`services.rs` (those two files' direct `policy::check` calls are pre-existing and explicitly out of scope for the voidwatch migration, per `voidwatch.rs`'s own doc comment).
- **Naming:** snake_case DB columns/Rust fields, kebab-case-ish REST paths, route path segments named after the resource (`:id`, `:name`, `:project_name`, `:host_id`, `:vmid`, `:user_id`, `:drive_id`, `:app_id` depending on domain).
- **Test conventions:** still sparse but growing. `#[test]`/`#[tokio::test]` now appear in `backend/src/api/tabs.rs`, `backend/src/api/agents.rs`, `backend/src/voidwatch.rs` (5 tests, `voidwatch.rs:164-279`), and `backend/src/api/integrations.rs` (new tests for `run_automation_job`, added alongside the P0-01 change). No established test-file-naming convention beyond `#[cfg(test)] mod tests { ... }` inline at the bottom of the same file as the code under test. A recurring pattern in the newer tests: since `db::run_migrations` doesn't create `policy_rules` (that happens in `db::init_pool`, outside the migration path tests use), test setup functions create it directly with an inline `CREATE TABLE` (see `voidwatch.rs:112-139`, referenced by comment from `integrations.rs`'s new tests).
- **Secrets:** encrypted with AES-256-GCM using a 32-byte key generated/stored at `<config_dir>/secrets.key` (0600 perms); the same decrypt routine is duplicated in `api/proxmox.rs` rather than shared — unchanged, still a candidate for consolidation but not something to change without a task naming it.
- **CLI subcommands** (`main.rs`) exist in parallel to the HTTP API for `user` and `backup` management, sharing the same `backend::auth`/`backend::backups` functions as the web handlers.
- **Core-logic-vs-HTTP-handler split, newly established by this batch:** `wireguard.rs`'s `add_peer`/`delete_peer` handlers were split into thin HTTP wrappers around new `pub(crate) add_peer_core`/`delete_peer_core` functions specifically so `node_enroll.rs` could reuse the exact provisioning logic without duplicating it or going through an HTTP round-trip. This is the first instance of that split pattern in the codebase (every other cross-handler reuse, e.g. `mcp.rs`/`studio.rs` sharing `invoke_tool`, was already function-level, not handler-level) — worth following if a future task needs one handler's logic from another.
- **Frontend:** one shared typed API client per client surface (`frontend/src/api/client.ts`, `mobile/src/api/client.ts` as an explicit port of it) namespaced by backend domain, `credentials: 'include'` (web) on every request; `frontend/`'s client additionally branches on `isTauri()` to use `@tauri-apps/plugin-http`'s fetch instead of the browser's when running inside the desktop wrapper (§5). Two parallel UI shells exist within `frontend/` (`frontend/src/pages/*` classic pages and `frontend/src/aios/panels/*`), plus now two additional, independent client trees (`frontend/src-tauri/`, `mobile/`) that are thin/native wrappers rather than forks of the React app — new panel work should check which shell *and* which client tree the task actually targets before choosing a template.
