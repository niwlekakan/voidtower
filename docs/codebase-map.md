New agents should not assume a top-level module's existence implies (or disproves) working logic — verify against the `api/` file and the module body itself.

## 4. AI-reachable ingress points (exhaustive, verified in source)

This is the most safety-critical section. Every claim below cites the exact call site. **Unchanged from the prior map's own findings** — re-verified directly rather than assumed, since nothing in `main`'s tracked history moved. The scope-bypass gap (bearer-token scoping) remains fully open in `main` — its accepted ADR (ADR-003) now has a reviewed, review-approved-on-the-merits implementation that was nonetheless rejected for carrying an unrelated forbidden-zone-contaminating reformat, and a further unmerged retry exists only on a local branch (see the preamble above). P0.4's hardcoded irreversibility denylist is still written and pushed but **not merged**, blocked on the same unresolved item-10 ambiguity.

### The voidwatch choke point (P0-01 → P0-02 → P0-03) — what each phase actually changed

`voidwatch::evaluate(db, actor, action_kind, action, resource) -> Verdict` (`voidwatch/mod.rs:132-159`) is the sole caller of `policy::check` from the three AI/automation-reachable ingress files. Enforced by a source-grep regression test (`voidwatch/mod.rs:533-549`) that fails the build if `api/mcp.rs`, `api/studio.rs`, or `api/integrations.rs` ever call `policy::check(` directly. Verified as true today for all three files.

**P0-01** was a no-op-by-default choke point: `Read`-classified actions always `Allow`; `Mutating` actions got exactly the same `policy::check` verdict as before. Its only effect was structural — `mcp.rs`/`studio.rs` started consulting `policy_rules` at all.

**P0-02 ended the no-op.** `policy.rs`'s `check()` now default-**denies** when no `policy_rules` row matches, for `actor_type` in `api_token` / `automation` / `ai` (`user` stays RBAC-governed, default-allow). Escape hatch: `voidwatch_default_allowlist` (§3), backfilled once on upgrade by `voidwatch::allowlist_seed::seed_default_allowlist_if_empty` (§1).

**P0-03 adds a pre-pass in front of both of the above** (`mode_pre_pass`, `voidwatch/mod.rs:165-193`), evaluated first, for *every* actor class equally (ADR-004 constraint 2 — the ladder isn't AI-only):
- Looks up the current `mode::Mode` for the resource's scope (falling back to the `"global"` scope row, §3). **If nothing has ever been configured — the default state of every install today, since no HTTP handler can set a mode yet (see §1's `mode.rs` entry) — the pre-pass is a complete no-op and falls straight through to P0-01/P0-02 behavior exactly as before.** This is why every ingress point below still behaves identically to the prior map's description in practice, despite the new code being live.
- `Observer` → `Deny` unconditionally. `Assisted` → `RequireApproval` unconditionally. `Yolo` → `Allow` unless the action's `risk_class::for_action()` is `Irreversible`, in which case `RequireApproval` (the "irreversibility denylist exception," coordinated with — but not dependent on — P0-04's not-yet-merged explicit list; see below). `Trusted` → `RequireApproval` for `Destructive`/`Irreversible`-risk actions always; for `Mutate`-risk actions, `Allow` only if `(actor_type, action, resource_type)` is on `voidwatch_default_allowlist` (same table P0-02 introduced, reused rather than duplicated) — allowlisted actions against a `SNAPSHOT_CAPABLE_RESOURCE_TYPES` resource (`vm`/`container`/`app`) get `AllowRequireSnapshot` instead of a plain `Allow`.
- `Verdict::AllowRequireSnapshot(String)` and `Verdict::RequireApproval(String)` are new terminal states (§1). **Neither ingress point below has anywhere to act on them today** — no approvals queue exists, and neither `mcp.rs`/`studio.rs`'s tool dispatch nor `integrations.rs`'s webhook path has a snapshot mechanism wired in — so both call sites treat every non-`Allow` verdict as a block (fail closed), documented in detail at `integrations.rs:726-735`'s `verdict_block_reason()` and applied identically by `mcp.rs`'s caller. **This was a live bug for one release**: `d378bb1` (same-day follow-up to `231ba2d`) fixed `integrations.rs`'s `webhook()` structured-action branch, which had silently treated `AllowRequireSnapshot` as `Allow` — reachable there (unlike `run_automation_job`'s fixed `"automation_job"` resource type) because `container` is snapshot-capable, so a Trusted-mode-mandated pre-apply snapshot was being skipped for `container.restart/start/stop`. Fixed with a shared helper specifically so the two call sites' handling can't drift apart again.

**Practically, as of this map:** unless an operator has explicitly configured a mode (no UI/API path to do so exists yet), every ingress point below behaves exactly as the P0-02-era map described. The mode ladder is live code, correctly tested (`voidwatch/mod.rs`'s `exhaustive_mode_by_risk_class_matrix` test enumerates all 16 mode×risk×allowlisted combinations), but dormant until wired to an operator-facing control.

### Risk classification (P0-03, `voidwatch/risk_class.rs`)

Two independent tables (§1) — `for_action()` is the one the mode ladder above actually consults; `ROUTE_RISK_CLASSES` is a separate, exhaustively-tested-but-not-yet-consumed per-route ledger intended as the classification P0-04's denylist will hang off. **`for_action()`'s fail-safe default is `Irreversible` for any action name it hasn't explicitly classified** — today this is the *only* enforcement of irreversibility-style caution in `main` (P0-04's purpose-built hardcoded list, `voidwatch/denylist.rs`, is not merged — see below), and only takes effect once a mode is configured (per the dormancy note above).

### Secrets redaction on the AI context path (P0.5, `api/redact.rs`, declared as `api::mcp::redact`)

New defense-in-depth layer, independent of and downstream from the voidwatch gating above — it runs on the *response*, not the request, and applies regardless of verdict (i.e. it also redacts an already-authorized `Allow` response). Two layers (module doc comment, `redact.rs:6-16`): (1) `redact_known_values` — exact-match redaction of every currently-stored `secrets` table value, decrypted via the same `api::secrets::decrypt` routine the Secrets panel uses; (2) `redact_patterns` — heuristic redaction of secret-*shaped* substrings not registered as VoidTower secrets at all (keyword=value pairs for a fixed keyword list including `password`/`api_key`/`token`/`credential`/etc., PEM private-key blocks, `Bearer <token>` headers, AWS `AKIA...` access keys), deliberately narrow (requires a keyword+separator or a structural marker) so hashes/UUIDs survive untouched (`redaction_does_not_break_non_secret_content` test). `redact_for_ai(state, text)` is the convenience entry point that fetches current known values and runs both layers. **Wired into exactly three places**, matching the module doc comment's claimed scope:
- `api/mcp.rs`'s `invoke_tool()` (`mcp.rs:573`) — every MCP tool-call result, before it's returned.
- `api/studio.rs`'s `mcp_invoke` — not a separate call; inherited automatically by reusing `mcp::invoke_tool` (§2), proven by a dedicated same-file test.
- `api/ai_context.rs`'s `get_context` (`ai_context.rs:159`) — the serialized context bundle.

Not wired into `api/ai_ask.rs` (`POST /api/ai/ask`) directly — that handler's own response isn't a tool-output/context-bundle passthrough in the same sense; verify against source before assuming redaction coverage there if a future task touches it.

One item (#10 of 12, "last remaining snapshot/backup of a resource") is escalated as a genuine textual ambiguity in ADR-004 itself — whether it means the last `backup_configs` schedule row (`api/backups.rs`, config coverage, not data loss) or the last Proxmox VM snapshot (`api/proxmox.rs`, actual data loss) — and is still parked, unanswered, in `.devteam/blocked/P0-04-irreversibility-denylist.md` as of this map. ### The scope-bypass gap (still open in `main` — accepted ADR, a reviewed-sound implementation exists but was rejected for an unrelated contamination defect)

The prior maps' second finding — upstream of `policy.rs`/`voidwatch/mod.rs` entirely — remains **unchanged in source** (verified: no diff to `bearer_auth.rs` or the relevant `auth/mod.rs` sections since `faf9d6b`, and `backend/src/auth/scope_enforce.rs` named in ADR-003's grant still does not exist in the tree). P0-02's `ai` actor class and P0-03's mode ladder are both real steps toward mitigating what this gap can reach, but neither is the fix — no ingress point constructs `ActorKind::Ai` yet, and the mode ladder gates *actions*, not the underlying identity-confusion bug:

1. `bearer_auth::middleware` (`bearer_auth.rs:10-27`) is mounted globally. A request with `Authorization: Bearer <token>` and no session cookie calls `auth::validate_api_token_any()` — checks only that the token hash exists and isn't expired, **never scopes** — and mints a full 1-hour session via `auth::create_temp_session()` for the token's owning user.
2. Every downstream handler's `require_user`/`require_admin`/`require_owner` helper then sees an ordinary session and applies the token owner's actual role, not the token's declared scope.
3. `ALL_SCOPES` (`integrations.rs:29-55`) is consulted in exactly two places in the whole codebase (`events.rs`, `integrations.rs`'s own `event_stream`). Every other scope-implying endpoint performs no scope check at all.
4. Net effect: a token minted with only `metrics:read`, if owned by an admin/owner user, can call any admin/owner-gated endpoint.

**What's new since the prior map:** the worker the prior map caught mid-claim finished the task and pushed `devteam/P0-06-scope-token-and-bearer-auth-bypass` — a single mandatory `scope_enforce::middleware` layer (per ADR-003's Decision, not a per-handler patch), an exhaustive deny-by-default `ROUTE_SCOPES` table, and capability-tier token minting. Adversarial review (`.devteam/logs/P0-06-scope-token-and-bearer-auth-bypass.review.md`) found the fix itself **sound** — all 7 named acceptance tests pass, the no-op-for-humans property is exercised end-to-end via a real `POST /api/auth/login`, `cargo test`/clippy/gates all green — but **rejected the PR** because the same commit also carried an unscoped `cargo fmt` reformat of `backend/src/api/apps.rs` that the review reproduced as cascading, via `rustfmt`'s `mod`-graph traversal, into two hard forbidden-zone files (`backend/src/oidc.rs`, `backend/src/api/auth.rs`) with no ADR covering them. The task is now in `.devteam/failed/`. A further retry (local branch `devteam/p0-06-work`, latest commit `4aa7965`, titled around "protect route table from rustfmt reflow") exists but has not been merged or even pushed to `origin` as of this map, and is not part of `main`'s history. **`main` itself is unaffected by any of this** — `backend/src/auth/scope_enforce.rs` does not exist in this tree, and `bearer_auth.rs`/`auth/mod.rs` are byte-identical to the prior map's survey. **Do not treat this section as current without re-checking `.devteam/active/`, `.devteam/blocked/`, `.devteam/failed/`, and `git branch --all` directly** — this is exactly the kind of state that changes independently of source and faster than this document can track it.

Also relevant to this gap: `node_enroll.rs`'s heartbeat token (§2/§3) was deliberately designed **not** to be an `api_tokens`-table token specifically because of this bypass — see the doc comment at `node_enroll.rs:64-68` — so the fleet-enrollment feature is not itself a new instance of the gap, it was built defensively around the known-open one.

## 5. Frontend panel → API dependency map

Frontend root: `frontend/src/`. Pages live in `frontend/src/pages/*.tsx`; a parallel "AIOS" native-panel system lives in `frontend/src/aios/panels/*.tsx` for an alternate embedded-desktop UI shell. Both consume a single shared API client, `frontend/src/api/client.ts`, organized as one `export const api = {...}` object with per-domain namespaces each calling a fixed `/api/...` path via a shared `request()` helper that sends `credentials: 'include'` and throws `ApiClientError` on non-2xx.

**Two additional, independent client surfaces now exist alongside `frontend/`, both new since the prior map:**

**Member-role gating (new, `frontend/src/` only):** `App.tsx`'s `MemberGate` (`App.tsx:60-71`) redirects any `role === 'member'` session to `/` for every path except `/` and `/apps`; `Sidebar.tsx` separately hides all infra nav for `member` users, replacing it with a two-item "Home"/"My Apps" group (`Sidebar.tsx:432-443`) — both client-side conveniences, not a security boundary (the real enforcement is server-side: `members.rs`'s handlers and `apps.rs`'s member-branching, §1/§2).

## 6. Observed code conventions

- **Auth pattern (repeated per-file, not shared middleware):** nearly every `api/*.rs` file defines its own local `require_user`/`require_admin`/`require_owner` async fn that reads the `vt_session` cookie from `CookieJar`, calls `auth::validate_session`, and maps `None`/error to `AppError::Unauthorized`/`Internal`. Role checks are inlined per-handler rather than expressed as a reusable Axum extractor. **The two new handler files (`members.rs`, `node_enroll.rs`) follow this exact same copy-per-file pattern** rather than introducing a shared helper — consistent with the existing convention, not a new one.
- **Global middleware layering:** now **three** cross-cutting concerns are implemented as actual Axum middleware (was two): `security_headers`, `demo_guard::middleware` (new), and `bearer_auth::middleware`, layered in that source order (`mod.rs:430-432`) — `.layer()` wraps outward, so the **last**-added layer is outermost and runs first on a request: `bearer_auth` (outermost/first) → `demo_guard` → `security_headers` → `cors` (innermost, closest to the router). This ordering is load-bearing: a bearer-token request has its temp session cookie minted by `bearer_auth` before `demo_guard` ever inspects the request for a `role = "demo"` session. CORS is permissive (`Any`/`Any`/`Any`).
- **Error handling:** single `AppError` enum via `thiserror`, with `#[from]` conversions for `sqlx::Error` and `anyhow::Error`; handlers return `Result<Json<T>>` and propagate with `?` or explicit `.map_err(...)`.
- **DB access:** raw `sqlx::query`/`query_as::<_, T>` with `#[derive(sqlx::FromRow)]` structs defined inline in the handler file (not a separate repository/DAO layer) — the new `members.rs`/`node_enroll.rs` continue this (e.g. `NodeRow` in `node_enroll.rs:74-85`, `DriveSummary` in `members.rs:71-79`).
- **Audit logging:** mutating human-initiated actions call `audit::log(...)` (`backend/src/audit/mod.rs:20-52`, §1); AI/automation-sourced actions use the sibling `audit::log_sourced(...)` with an extra `source` param (e.g. `"odysseus"`). Every new `members.rs`/`node_enroll.rs` mutating handler follows this convention (e.g. `members.access.grant`, `nodes.enroll`, `nodes.pairing_code.create`).
- **Actor detection for policy/voidwatch:** the `MaybeTokenActor` Axum extractor (`policy.rs`) reads an `ApiTokenActor` marker inserted by `bearer_auth::middleware`; ingress handlers construct a `voidwatch::Actor{ kind }` explicitly per call site (`ActorKind::ApiToken` in `mcp.rs`'s bearer-token dispatch, `ActorKind::User` in `studio.rs`'s session-authenticated path, `ActorKind::Automation` in `integrations.rs`'s webhook paths) rather than deriving it generically — this is opt-in/explicit per-handler, not automatic, same spirit as the pre-existing `if is_token { policy::check(...) }` pattern in `containers.rs`/`services.rs` (those two files' direct `policy::check` calls are pre-existing and explicitly out of scope for the voidwatch migration, per `voidwatch/mod.rs`'s own doc comment).
- **Naming:** snake_case DB columns/Rust fields, kebab-case-ish REST paths, route path segments named after the resource (`:id`, `:name`, `:project_name`, `:host_id`, `:vmid`, `:user_id`, `:drive_id`, `:app_id` depending on domain).
- **Test conventions:** still sparse but growing. `#[test]`/`#[tokio::test]` now appear in `backend/src/api/tabs.rs`, `backend/src/api/agents.rs`, `backend/src/voidwatch/mod.rs` (17 tests including P0-03's mode-ladder acceptance suite, `voidwatch/mod.rs:249-934`), `backend/src/voidwatch/mode.rs` and `backend/src/voidwatch/risk_class.rs` (each their own test module, new in P0-03), `backend/src/voidwatch/allowlist_seed.rs`, `backend/src/policy.rs` (`#[cfg(test)] mod tests`, added alongside P0-02, including a data-driven "default verdict by actor class" table), `backend/src/api/integrations.rs` (tests for `run_automation_job`/`webhook`, extended by P0-03's `d378bb1` fix with an end-to-end `AllowRequireSnapshot`-blocks regression test), and `backend/src/api/mcp.rs`/`backend/src/api/redact.rs`/`backend/src/api/studio.rs` (P0.5's redaction corpus tests — a fixed set of known-secret-shaped strings asserted absent from tool-call/context-bundle output end-to-end). No established test-file-naming convention beyond `#[cfg(test)] mod tests { ... }` inline at the bottom of the same file as the code under test. A recurring pattern in the newer tests: since `db::run_migrations` doesn't create `policy_rules`/`voidwatch_default_allowlist`/`voidwatch_mode_settings` (those live in `db::init_pool`, outside the migration path tests use), test setup functions create all three directly with inline `CREATE TABLE` statements — `voidwatch::tests::create_policy_tables` (`voidwatch/mod.rs:259-305`) is `pub(crate)` specifically so `voidwatch::mode`'s own test module can reuse it rather than duplicating the statements. Also: `backend/tests/schema_golden.sql` plus a `db/mod.rs` test (`schema_golden_file_matches_live_schema_after_migration`, `db/mod.rs:752-`) that diffs live `sqlite_master` schema for both voidwatch tables against it, and a sibling test (`pre_existing_tables_byte_identical_after_upgrade`, `db/mod.rs:812-`) proving `init_pool` adds new tables to a simulated pre-upgrade database without dropping/narrowing anything pre-existing — a heavier-weight regression style than anything in the codebase before P0-02, worth following for future schema-adjacent grants.
- **Parser-based exhaustiveness tests (new pattern, P0-03):** two source-grep-style regression tests now go further than string-matching — `risk_class::tests::every_registered_route_has_a_risk_class` (`risk_class.rs:677-717`) parses `api/mod.rs` with a manual balanced-paren scanner to extract every registered `(method, path)` and asserts a bijection against `ROUTE_RISK_CLASSES`, and `voidwatch::mod::tests::evaluate_is_the_only_caller_of_policy_check_from_ai_ingress` (`voidwatch/mod.rs:533-549`) is the substring-based precedent it extends. Same spirit as `db/mod.rs`'s schema-golden test: catch a class of drift (unclassified route, ungated ingress point, schema divergence) at build/test time rather than relying on a human noticing in review.
- **Shared-reasoning-helper-to-prevent-drift (new pattern, established by the `d378bb1` bugfix):** `integrations.rs`'s `verdict_block_reason()` (`integrations.rs:726-735`) exists because two near-identical inline `match` arms at separate call sites drifted apart — one was correct for its fixed resource type, the other silently inherited wrong reasoning via copy-paste. The fix was to make both call sites share one function rather than just fixing the one that was wrong. Worth applying this pattern proactively anywhere near-identical verdict/permission-handling logic is about to be copy-pasted to a second call site.
- **`test_support.rs` as a per-module shared-fixture file (new, P0.5):** `api/mcp.rs` declares `pub(crate) mod test_support;` (backed by `api/test_support.rs`, `#![cfg(test)]`-gated) providing a shared `AppState` builder (`build()`) and session/DB setup helpers (`setup_db()`, `user_with_session()`) so `redact.rs`'s and `mcp.rs`'s own test modules don't each hand-roll `AppState` construction. First instance of a dedicated fixture file in this codebase — prior test modules each inlined their own setup helpers locally (e.g. `voidwatch::tests::setup_db` above is still local, not factored out this way) — worth reusing rather than re-duplicating if a future task's tests need the same `AppState` boilerplate.
- **Secrets:** encrypted with AES-256-GCM using a 32-byte key generated/stored at `<config_dir>/secrets.key` (0600 perms); the same decrypt routine is duplicated in `api/proxmox.rs` rather than shared — unchanged, still a candidate for consolidation but not something to change without a task naming it.
- **CLI subcommands** (`main.rs`) exist in parallel to the HTTP API for `user` and `backup` management, sharing the same `backend::auth`/`backend::backups` functions as the web handlers.
- **Core-logic-vs-HTTP-handler split, newly established by this batch:** `wireguard.rs`'s `add_peer`/`delete_peer` handlers were split into thin HTTP wrappers around new `pub(crate) add_peer_core`/`delete_peer_core` functions specifically so `node_enroll.rs` could reuse the exact provisioning logic without duplicating it or going through an HTTP round-trip. This is the first instance of that split pattern in the codebase (every other cross-handler reuse, e.g. `mcp.rs`/`studio.rs` sharing `invoke_tool`, was already function-level, not handler-level) — worth following if a future task needs one handler's logic from another.
- **Frontend:** one shared typed API client per client surface (`frontend/src/api/client.ts`, `mobile/src/api/client.ts` as an explicit port of it) namespaced by backend domain, `credentials: 'include'` (web) on every request; `frontend/`'s client additionally branches on `isTauri()` to use `@tauri-apps/plugin-http`'s fetch instead of the browser's when running inside the desktop wrapper (§5). Two parallel UI shells exist within `frontend/` (`frontend/src/pages/*` classic pages and `frontend/src/aios/panels/*`), plus now two additional, independent client trees (`frontend/src-tauri/`, `mobile/`) that are thin/native wrappers rather than forks of the React app — new panel work should check which shell *and* which client tree the task actually targets before choosing a template.
