# VoidTower Codebase Map

Generated by a read-only survey of `niwlekakan/voidtower` @ `main`, cross-checked line-by-line against source. This map is descriptive, not authoritative — verify behavior in source before relying on it for policy/auth/schema work. All paths are relative to the repo root unless stated otherwise.

## 1. Backend module inventory

Modules declared in `backend/src/main.rs:1-21` plus `backend/src/api/mod.rs:32-80`.

| Module | Purpose (verified in source) |
|---|---|
| `backend/src/main.rs` | Binary entry point: CLI (clap) with `user`/`backup` subcommands, `AppState` definition (`main.rs:39-51`), DB init, background loops (metrics collector, alert thresholds, status-check scheduler, automation scheduler, restore-test scheduler, session cleanup, Proxmox VM monitor), router mount, static frontend serving. |
| `backend/src/ai/mod.rs` | Multi-provider AI abstraction: `AiProvider` trait (`complete`/`stream`/`health_check`), `ProviderConfig` persisted in `ai_providers` table, re-exports `AiOrchestrator` and `build_provider`. Submodules `providers/` (odysseus, openai, anthropic, local) and `orchestrator.rs`/`router.rs`. |
| `backend/src/alerts/mod.rs` | Stub only — comment-only placeholder (`alerts/mod.rs:1-8`) describing planned `evaluate_alerts`/`acknowledge_alert` functions. **Actual alert logic lives in `backend/src/api/alerts.rs`**, not here. |
| `backend/src/auth/mod.rs` | Real auth core: `User`/`Session` structs, Argon2id password hashing, session create/validate (`validate_session` at `auth/mod.rs:256`), API token validation both scoped (`validate_api_token`, `auth/mod.rs:334`) and unscoped (`validate_api_token_any`, `auth/mod.rs:292`), temp-session minting for token requests (`create_temp_session`, `auth/mod.rs:321`), role management. |
| `backend/src/automation/mod.rs` | Stub only — comment-only placeholder (`automation/mod.rs:1-5`) for a future YAML automation DSL. **Actual automation job CRUD/execution lives in `backend/src/api/automation.rs`.** |
| `backend/src/backups/mod.rs` | Restic wrapper: `BackupConfig` struct, `is_restic_available()`, run/check/restore-test functions (used by both `api/backups.rs` and the CLI `backup` subcommand in `main.rs`). |
| `backend/src/cluster/mod.rs` | Stub only (`cluster/mod.rs:1-9`) — just an `is_agent_mode()` helper checking for `--agent` CLI flag; no actual multi-node logic implemented. |
| `backend/src/config/mod.rs` | `Config` struct (bind/port/data_dir/frontend_dir/etc.), loaded from file + env, includes the `host_data_dir` vs `data_dir` distinction documented at `config/mod.rs:17-28` for containerized VoidTower deployments. |
| `backend/src/containers/mod.rs` | Docker/bollard wrapper: list/action/logs/exec on containers, plus `DeployRegistry` (`containers/mod.rs:19`) for tracking in-flight `docker compose` deploy PIDs so they can be cancelled gracefully. |
| `backend/src/db/mod.rs` | All schema definition: `init_pool()` runs `run_migrations()` (baseline `CREATE TABLE` block) then a long, ad-hoc sequence of `CREATE TABLE IF NOT EXISTS` / `ALTER TABLE ADD COLUMN` statements accreted over time (`db/mod.rs:24-179`). No sqlx migration files exist (`backend/migrations/` does not exist — verified via glob). |
| `backend/src/error.rs` | `AppError` enum (`NotFound`, `Unauthorized`, `Forbidden`, `PolicyDenied`, `BadRequest`, `Conflict`, `FeatureUnavailable`, `TooManyRequests`, `TotpRequired`, `Database`, `Internal`) with a single `IntoResponse` impl mapping each to `(StatusCode, error_code, message)` JSON. |
| `backend/src/monitoring/mod.rs` | `MetricsSnapshot`/`DiskInfo`/`NetworkInfo`/`ProcessInfo`/`GpuInfo` structs and the `MetricsCollector`/`MetricsBroadcaster` used by `main.rs`'s metrics loop and `api/metrics.rs`. |
| `backend/src/networking/mod.rs` | Stub only (`networking/mod.rs:1-8`) — comment describing planned interface/route/firewall introspection. **Actual network + firewall logic lives in `api/network.rs` and `api/firewall.rs`.** |
| `backend/src/oidc.rs` | OIDC/Authentik SSO: loads per-instance `OidcSettings` from DB (`oidc.rs:28`), PKCE flow via the `openidconnect` crate, role mapping from claims. |
| `backend/src/policy.rs` | The AI/automation policy engine (see §4 for detailed analysis). `PolicyRule` struct, `check()` function (default-allow when no rule matches — `policy.rs:36-81`), `ApiTokenActor`/`MaybeTokenActor` extractor used by handlers to detect token-originated requests. |
| `backend/src/security/mod.rs` | Stub only (`security/mod.rs:1-5`) — comment describing planned security-scan checks. **Actual session-revocation logic lives in `api/security.rs`.** |
| `backend/src/services/mod.rs` | systemd wrapper: `ServiceInfo`, `ServiceAction` enum (Start/Stop/Restart/Enable/Disable), `is_systemd_available()`, shells out to `systemctl`. |
| `backend/src/storage/mod.rs` | Block device / mount / RAID / SMART introspection structs (`BlockDevice`, `MountInfo`), used by `api/storage.rs`. |
| `backend/src/terminal/mod.rs` | PTY handling via `portable-pty` for the in-browser terminal (`ClientMessage`/`ServerMessage` WS protocol), used by `api/terminal.rs`. |
| `backend/src/vms/mod.rs` | Stub-ish: only `is_libvirt_available()` implemented (`vms/mod.rs:8-15`); comment describes planned `list_vms`/`vm_action`. Local KVM + Proxmox VM logic actually lives in `api/vms.rs` and `api/proxmox.rs`. |

**Convention note:** several top-level modules (`alerts`, `automation`, `cluster`, `networking`, `security`, and partially `vms`) are comment-only placeholders from an earlier phased plan; the real, shipped logic for those feature areas lives entirely in the corresponding `backend/src/api/*.rs` file. New agents should not assume a top-level module's existence implies working logic — verify against the `api/` file.

## 4. AI-reachable ingress points (exhaustive, verified in source)

This is the most safety-critical section. Every claim below cites the exact call site.

### Ingress points and their actual gating status

**`backend/src/api/mcp.rs` — MCP server (`GET /api/mcp`, `POST /api/mcp/message`)**
- Auth: `check_mcp_auth()` (`mcp.rs:80-99`) requires `odysseus.mcp_enabled` setting = `"true"` and a valid Bearer token via `auth::validate_api_token_any` (unscoped — any valid token from any user works).
- `tools/call` dispatch (`mcp.rs:267-279`) only exposes **read-only** tools: `list_nodes`, `get_node_metrics`, `list_containers`, `list_services`, `list_alerts`, `get_container_logs`, `list_routes`, `read_file`, `search_code`, `get_template`. **No mutating tool exists in this handler today**, and it never calls `policy::check`. Grep of the whole tree confirms `mcp.rs` has zero references to `policy::` (only 5 files in the repo call `policy::check`: `integrations.rs`, `containers.rs`, `services.rs`, `policy.rs` itself, and `main.rs` does not). So this endpoint is currently safe *by omission* (no write tools registered), not by policy enforcement — adding a mutating tool here would ship completely ungated, exactly as `docs/gap-analysis.md` §2 P0.1 warns.
- `tool_read_file`/`tool_search_code` (`mcp.rs:400-423`, delegating to `ai_context.rs:342-389`) do path-traversal sanitization (`rel.contains("..")` rejected, `full.starts_with(root)` checked) but otherwise expose the entire repo tree read-only to any valid token holder.

**`backend/src/api/studio.rs` `mcp_invoke` (`POST /api/studio/mcp/invoke`)**
- Auth: `require_user()` (session cookie only, `studio.rs:21-27`) — this is the in-UI "Studio" panel's tool-test button, not itself AI-agent-reachable over the network with a bearer token. It calls `super::mcp::invoke_tool()` (`studio.rs:563`), which is the **same read-only tool set** as `mcp.rs`'s `tools/call` (`mcp.rs:437-451`) reused verbatim. No `policy::check` call anywhere in `studio.rs` (grep confirms). Same "safe only because no write tools are registered" caveat applies.

**`backend/src/api/integrations.rs` `manifest` (`GET /api/integrations/odysseus/manifest`)**
- No auth check at all (handler takes only `State`, no session/token param — `integrations.rs:472`). It **documents** a tool surface including mutating tools (`restart_service`, `restart_container`, `deploy_app`, `run_backup`, `run_automation`, `acknowledge_alert` — `integrations.rs:502-516`) with `required_scope` fields, but this is only a JSON description consumed by Odysseus; it does not itself execute anything. The actual execution paths it documents (`POST /api/services/:name/action`, `POST /api/containers/:id/action`, etc.) are the real ingress points and are analyzed under "the scope-bypass gap" below.

**`backend/src/api/integrations.rs` `webhook` (`POST /api/integrations/webhooks`)** — this IS gated, partially:
- Auth: constant-time comparison against `odysseus.webhook_secret` (`integrations.rs:699-708`), plus `odysseus.enabled`/`emergency_disabled` checks.
- Structured resource actions (`container.restart/start/stop`, `service.restart/start/stop`) **do call the policy choke point**: `policy::check(&state.db, "automation", action_name, resource_type, &resource_id)` at `integrations.rs:819`, and a `Deny` verdict returns `AppError::PolicyDenied` (`integrations.rs:820-828`). This is the one ingress point in the whole codebase that matches the EDD's intended shape.
- **However**, the `automation_id`-triggered path in the same handler (`integrations.rs:712-801`) is **not policy-gated at all**: it looks up an `automation_jobs` row by ID (only checking `enabled = 1`), then runs `job.command` as an arbitrary shell command via `tokio::process::Command::new("bash").arg("-c").arg(&command)` (`integrations.rs:753-757`) with no call to `policy::check`. Any user with the webhook secret (a single shared value, same secret used for all triggers) can execute any pre-registered automation job's shell command, and — separately — any authenticated UI user can create arbitrary-command automation jobs via `POST /api/automation` (`api/automation.rs:65-83`, only requires role != "viewer"). This is a materially larger blast radius than the resource-action path and is not mentioned as gated anywhere in the gap analysis.

**`backend/src/api/ai_ask.rs` (`POST /api/ai/ask`)**
- Auth: session cookie only (`ai_ask.rs:30-37`) — this is the in-UI chat assistant, human-initiated. It streams to either the multi-provider orchestrator or the legacy Odysseus URL and does not itself expose or execute tools; it returns a proxied model response. The embedded system prompt (`ai_ask.rs:155-183`) documents an **external** MCP tool server (`odysseus-mcp-servers/voidtower_server.py`, not in this repo) exposing write tools (`vt_control_service`, `vt_control_container`, `vt_control_app`, `vt_deploy_app`, `vt_control_vm`, etc.) that call the same underlying REST endpoints (`/api/services/:name/action`, `/api/containers/:id/action`, etc.) — meaning the real ingress control point for those write tools is the underlying REST handler's own auth, not `ai_ask.rs`.

**`backend/src/api/webhooks.rs` (`/api/webhooks*`)**
- This is the **outbound** notification-webhook system (ntfy/Discord/Slack/generic), session/admin-gated (`require_admin`, `webhooks.rs:21-34`), used to notify humans of alerts — not an AI ingress point at all. Do not confuse with `integrations.rs`'s inbound `webhook` handler (same name, opposite direction).

**`backend/src/api/ai.rs`, `ai_context.rs`, `ai_providers.rs`, `agents.rs`**
- `ai.rs` (llama.cpp process status/kill) and `ai_providers.rs` (AI provider CRUD) are both `require_admin`-gated (session only), configuration surfaces, not AI-callable action endpoints.
- `ai_context.rs`'s `get_context` (`ai_context.rs:80`) is `require_any_user`-gated and read-only (dumps repo structure/conventions for the in-UI assistant); its `read_project_file`/`search_project_code` helpers are reused by `mcp.rs` (see above).
- `agents.rs` is the dev-team agent-status dashboard (unrelated to Odysseus/AI actions) — `require_user`/`require_admin` session-gated throughout, with a `post_status` ingest endpoint (`agents.rs:413`) intended for dev agents to self-report status, gated by `require_admin`.

### The scope-bypass gap (the largest finding, not previously documented in gap-analysis.md)

`docs/gap-analysis.md` describes the policy problem as "flat deny-rules, default-allow, 3 call sites." Verified in source, there is a second, more fundamental gap upstream of `policy.rs` entirely:

1. **`bearer_auth::middleware`** (`backend/src/api/bearer_auth.rs:10-27`) is mounted globally on the entire `main_router` (`api/mod.rs:412`), covering essentially every route in the table above except `/api/health` and the login endpoints.
2. When a request carries `Authorization: Bearer <token>` and no session cookie, the middleware calls `auth::validate_api_token_any()` (`auth/mod.rs:292-318`) — **which does not check scopes at all**, only that the token hash exists and isn't expired — and on success mints a full 1-hour session via `auth::create_temp_session()` (`auth/mod.rs:321-332`) for the token's owning user, injecting a `vt_session` cookie into the request (`bearer_auth.rs:17-24`).
3. From that point on, every downstream handler's `require_user`/`require_admin`/`require_owner` helper (a pattern repeated per-file, e.g. `containers.rs:46-55`, `services.rs:118`, `system.rs:9`, `disaster.rs`) sees a normal, valid session and applies **the token owner's actual role** (owner/admin/operator/viewer) — not the token's declared scopes.
4. The `ALL_SCOPES` list in `integrations.rs:29-55` (`metrics:read`, `containers:restart`, `apps:deploy`, `secrets:list`, etc.) is enforced in exactly **two** places in the entire codebase: `auth::validate_api_token()` (the scoped variant) is called only from `events.rs:49/58` (`GET /api/events/stream`) and `integrations.rs:551/560` (`GET /api/integrations/events`). Every other endpoint that a scope name implies restricting (e.g. `apps:deploy` → `POST /api/apps/deploy`, `secrets:list` → `GET /api/secrets`) performs no scope check whatsoever.
5. **Net effect:** an API token created with only the `metrics:read` scope, if it belongs to an owner/admin user (the only roles allowed to create tokens per `require_admin` in `integrations.rs:94-107`), can call `POST /api/system/update`, `GET /api/secrets/:id/reveal`, `POST /api/storage/format`, `POST /api/disaster/emergency-reset-admin`, or any other admin/owner-only endpoint — because the middleware upgrades any valid token into a full session of that user's role, and scopes are simply never consulted by those handlers. The policy.rs `actor_type = "api_token"` check inside `containers.rs`/`services.rs` (`policy::check(&state.db, "api_token", ...)`, only invoked `if is_token`) is the *only* additional restriction on token-originated container/service actions, and per §2 of the gap analysis it is default-allow with 3 seeded deny rules.

This is a materially bigger and more concrete finding than the "MCP tools/call is ungated" line in `gap-analysis.md` §1, because it means the scoped-token system displayed throughout the Integrations UI is currently decorative for the vast majority of the API surface, not just the MCP/Studio paths gap-analysis names. P0.6 ("scope down VOIDTOWER_TOKEN... mode ladder + scopes compose") cannot land as described without first fixing this upstream bypass in `bearer_auth.rs`/`auth::validate_api_token_any`.

## 6. Observed code conventions

- **Auth pattern (repeated per-file, not shared middleware):** nearly every `api/*.rs` file defines its own local `require_user`/`require_admin`/`require_owner` async fn (e.g. `containers.rs:46-55`, `services.rs:118-124`, `webhooks.rs:21-34`, `system.rs:9`, `agents.rs:6-11`) that reads the `vt_session` cookie from `CookieJar`, calls `auth::validate_session`, and maps `None`/error to `AppError::Unauthorized`/`Internal`. Role checks are inlined per-handler (`if user.role == "viewer" { return Err(Forbidden) }` or `!matches!(user.role.as_str(), "owner"|"admin")`) rather than expressed as a reusable Axum extractor — a new contributor should follow this copy-per-file pattern to match existing style, even though it is duplicative.
- **Global middleware layering:** only two cross-cutting concerns are implemented as actual Axum middleware: `security_headers` (`api/mod.rs:12-30`) and `bearer_auth::middleware` (`api/mod.rs:412`). CORS is permissive (`Any`/`Any`/`Any`, `api/mod.rs:83-86`).
- **Error handling:** single `AppError` enum (`error.rs:9-33`) via `thiserror`, with `#[from]` conversions for `sqlx::Error` and `anyhow::Error`; handlers return `Result<Json<T>>` (repo-local `Result` alias) and propagate with `?` or explicit `.map_err(AppError::Internal)`/`.map_err(AppError::Database)`.
- **DB access:** raw `sqlx::query`/`query_as::<_, T>` with `#[derive(sqlx::FromRow)]` structs defined inline in the handler file (not a separate repository/DAO layer) — e.g. `TokenRow` in `integrations.rs:136-145`, `AlertRow` inline in `mcp.rs:351-359`.
- **Audit logging:** mutating human-initiated actions call `audit::log(&state.db, user_id, "human", "<action>", resource_type, resource_id, outcome, ip, details)` (signature at `audit/mod.rs:22-32`); AI/automation-sourced actions use the sibling `audit::log_sourced(...)` with an extra `source` param (e.g. `"odysseus"` — see `integrations.rs:722-734`), which the `actor_type` column then makes queryable as `'agent'` rows (`integrations.rs:929`, `recent_actions`).
- **Actor detection for policy:** the `MaybeTokenActor` Axum extractor (`policy.rs:96-113`) reads an `ApiTokenActor` marker inserted into request extensions by `bearer_auth::middleware`, and handlers that want to gate token-originated mutations check `if is_token { policy::check(...) }` (`containers.rs:100-108`, `services.rs:82-88`) — this is opt-in per-handler, not automatic.
- **Naming:** snake_case DB columns/Rust fields, kebab-case-ish REST paths (`/api/apps/deploy-custom`), route path segments named after the resource (`:id`, `:name`, `:project_name`, `:host_id`, `:vmid` depending on domain).
- **Test conventions:** almost no tests exist. Grep confirms `#[test]`/`#[tokio::test]` appear only in `backend/src/api/tabs.rs` and `backend/src/api/agents.rs` (the latter has 4 tests at `agents.rs:492-627` using an in-memory SQLite pool seeded via `crate::db::run_migrations`). There is no established test-file-naming convention beyond `#[cfg(test)] mod tests { ... }` inline at the bottom of the same file as the code under test.
- **Secrets:** encrypted with AES-256-GCM using a 32-byte key generated/stored at `<config_dir>/secrets.key` (0600 perms) — see `main.rs:243-266`; the same decrypt routine is duplicated in `api/proxmox.rs:14-24` (comment: "same logic as api/secrets.rs") rather than shared, which is a candidate for consolidation but not something to change without a task naming it.
- **CLI subcommands** (`main.rs:104-189`) exist in parallel to the HTTP API for `user` and `backup` management, sharing the same `backend::auth`/`backend::backups` functions as the web handlers — useful for scripting without going through the API/auth layer.
- **Frontend:** one shared typed API client (`frontend/src/api/client.ts`) namespaced by backend domain, `credentials: 'include'` on every request, typed `ApiClientError` thrown on non-2xx; two parallel UI shells exist (`frontend/src/pages/*` classic pages and `frontend/src/aios/panels/*` an embedded "AIOS" desktop-style panel system with its own `PANEL_REGISTRY` in `AiosLayout.tsx` per the template at `ai_context.rs:238`) — new panel work should check which shell the task actually targets before choosing a template.
